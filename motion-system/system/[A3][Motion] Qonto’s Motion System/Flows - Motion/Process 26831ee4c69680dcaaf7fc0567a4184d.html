Process

What Your Engineers Need (Based on Their Requests)

  1. A single, easy-to-find source of truth for animation patterns
    • No more guessing or asking around for durations/styles/directions.
    • No more decisions left to chance or “whatever’s in Figma.”
  1. Clear standards for the most common animation types:
    • Change content (micro-interactions, feedback)
    • Object visibility (show/hide elements)
    • Screen transitions (flows, modals, onboarding, etc.)
    • Change screen state (loading, loaded, error, etc.)
  1. For each pattern:
    • Name of the animation (e.g., “Screen transition: Next step”)
    • Where to use it (with examples)
    • Precise definition of movement (direction), effect (fade, slide, scale, etc.)
    • Which tokens to use (duration, easing, delay, translate, alpha…)
    • Platform-specific notes (is this OS-managed? Do we override for brand reasons?)
    • Visual links (Figma/MP4 for reference)
    • Rationale (Why this direction? Why this effect? What is the user supposed to feel or understand?)
  1. Explicit “do not animate” guidance for cases where animation should be avoided.
  1. Consistency across teams (Android, iOS, Web)—so a feature isn’t animated differently each time.

What You Need to Build Next

1. Motion Patterns Library

  • For each of the core patterns (change content, object visibility, screen transition, screen state), create a Notion page or card with:
    • Pattern name
    • Description
    • Where to use
    • Tokens to use (duration, easing, direction, effect…)
    • Platform notes
    • Rationale (1–2 lines on why this is the right pattern for this case)
    • Visual example (Figma link, GIF, or screenshot)
    • “Do not use for…” (if any caveats)

2. Rationales for Each Pattern

  • Why do we use this direction and effect?(E.g., “Up = entering focused mode”; “Left = forward in flow”; “Fade = change of state, not space”)
  • What is the expected user feeling or mental model?(“Progress”, “Return”, “Completion”, “Alternative path”)

3. How to Roll Out

  • Start with the most common flows and patterns (flows, modals, banners, loaders).
  • Document one pattern fully as a model—get feedback from engineers and designers, then scale up.
  • Link each pattern to your token definitions and Figma/Notion visual reference.
  • Communicate to all teams where to find these docs and how to give feedback.

Summary Checklist

  • Rationales for direction/effect for every transition
