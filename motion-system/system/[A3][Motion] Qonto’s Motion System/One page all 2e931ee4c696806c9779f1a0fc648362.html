Motion System so far

Principles

Animations enhance user experience by adding interactivity and visual delight to our brand. They guide users, provide feedback, and make interactions feel more natural and engaging.

We divide our Motion Principles in 4 pillars:

When to use animation

Animation at Qonto is purposeful, it breathes life into our products, communicates meaning, and supports storytelling.
We use animation only when it adds genuine value, never for decoration or visual noise.


We animate to:

  • Guide users’ attention to what matters
  • Clarify interactions and improve usability
  • Express our brand’s personality through meaningful delight

Use cases

Animation usage at Qonto is organized by platform, distinguishing between Product and Website experiences.


Guidelines specify when motion is appropriate, what form it should take so that motion is applied purposefully and consistently, contributing to a unified and engaging user experience across all touch points.

What are Motion tokens?

Motion tokens are the foundation of a scalable Motion System at Qonto.
They are divided into two main groups:

Tokens help bridge design and development by using shared language and structure, enabling smoother handoff and easier maintenance.

Primitive tokens

Timing

Core time controls for any animation.

Values

Visual change primitives applied to elements.

Semantic tokens

Base

Purpose-driven combinations of timing primitives with no visual change.
Designers pick by intent - engineers apply effects as needed or use Common patterns.

Common

Ready-made patterns combining Base timing with effect(s).

Motion Decision Guide

Quick start: Pick Common tokens first - they cover most use cases. If you need something custom, combine a Base token with Values.

How to choose:

  • Start with Common tokens → Use the sections below to find your match.
  • Review the token details → Each token has Purpose, Use/Don't, Properties, Preview, and Code alias.
  • Customize if needed → Pair a Base token (entry, exit, reveal, transition)
    with Value primitives.

If no Common token fits your needs:

  1. Pick a Base token (motion-entry, motion-exit, motion-reveal, motion-transition)
  1. Combine it with Value primitives (motion-translate, motion-scale, motion-alpha, etc.)

Token Structure

Motion tokens follow a 3-tier hierarchy:

1. Primitives → motion-duration-200, motion-scale-95, motion-alpha-0

  • Building blocks for timing and visual changes

2. Base → motion-entry-default (duration + easing + delay)

  • Purpose-driven timing combinations (no visual change)

3. Common → motion-slide-up (Base + Values)

  • Ready-made patterns combining timing with effects

Current Animation Solution

Behavior

Why

  • Main header elements (title, filters, etc.) slide left and fade out.
  • Selection mode header (bulk actions, select all, count, etc.) slides in from the right and fades in.
  • Bottom bar: The bottom bar (e.g. navigation, filters, quick actions) slides down and fades out.
  • When exiting selection mode, the reverse animation occurs.

  • Directional movement reinforces the change in mode and visually links the header change to the appearance of selection indicators in the list.
  • The animation feels dynamic, yet remains subtle and professional, just smooth sliding and fading.
  • A slide down feels spatially connected to the screen layout, it reinforces that the bar is dismissed due to mode change.
  • Avoids confusion: a fade out could be misread as a visual glitch or passive disappearance.

New animation solution

Behavior

Focus only on the charts and numbers after the submit

When the user clicks Calculate, the result area (numbers and charts) animates into view.

  • Numbers (e.g. gains, savings) use a count-up animation from zero (or previous value) to the new result.
  • Charts fade in and animate (e.g. sweep or fill effect).
  • If the user edits an input, results do not update until the next submit.
  • If the widget is reset or recalculated, the animation sequence replays.

Why

Reduces distraction:

No live-updating numbers while typing, so focus remains on input.

Provides feedback:

Animation links the user’s action (button click) to the new results, reinforcing cause and effect.

Professional and calm:

Subtle, purposeful motion signals change without feeling “gimmicky.”

Clear state change:

Animation only after calculation makes the transition more meaningful.

Recommendation summary

Calculator Motion Guidelines

  • Default behavior
    All calculator widgets should animate numbers and charts only after the user submits.
  • Animated numbers
    Use count-up animation for numerical results only on submit, not live while typing.
  • Charts
    Charts should fade in and animate after submit, or when live updates are truly required.
  • Live updates
    Only use live-updating numbers or charts if the calculator is extremely simple, and real-time feedback is essential for usability.
  • No animation on input
    Avoid animating numbers or charts as the user types, unless there is a clear product need and it will not distract.

Website needs

Live edit, no button

Live text, controlled motion

Calculators should display results and update charts live as the user types but motion must remain subtle.

Charts

Charts should subtly animate (e.g., sweep or fill) when values change, but keep transitions fast and with minimal motion to prevent distraction.

Animated numbers

Use a quick, gentle count-up animation (optional: only if the value change is significant otherwise just swap numbers with a subtle fade).

New Brand Principles

Principles to Motion

Our Motion language

Intensity bands


Rationals

Derived them

Start from brand → product needs

  • Efficient: reduce cognitive load, be precise, “show don’t tell.”
  • Personal: responsive, acknowledges the user.
  • Elevated: polished, readable, never muddy.
  • Attentive: guides, keeps orientation.

Identify the recurring product problems motion must solve

  • What happened first/next?
  • Where did it come from/go to?
  • Can I still read this?
  • What should my eye follow?
  • Why does this feel jittery or messy?
  • Why did that just pop?

Cluster those problems into non-overlapping, reviewable rules

  • Each rule fixes a distinct failure mode.
  • Names must be neutral, brand-voiced, and usable in crits.
  • Keep the set small but complete; stackable across patterns.

The seven categories and the “failure mode” each addresses

  • Orchestration → solves “what happened first/next?” (sequence clarity)
  • Spatial continuity → solves “where did it come from/go to?” (context)
  • Clear fades → solves “I can’t read this/muddy overlap” (legibility)
  • Directional focus → solves “my eye is pulled everywhere” (attention)
  • Grid‑aligned movement → solves “diagonal drift/smart-animate artifacts” (precision)
  • Spacing discipline → solves “collisions/jank in dense UI” (order)
  • Flow → solves “things pop or jump abruptly” (continuity)

How they map back to the brand principles

  • Efficient: Directional focus, Grid‑aligned, Orchestration
  • Personal: Flow, Orchestration (acknowledging the user, not performing)
  • Elevated: Clear fades, Spacing discipline (polish you feel more than see)
  • Attentive: Spatial continuity, Orchestration, Flow (guidance and orientation)

Product Motion Principles

Product Motion Language

Product Motion Behaviors

  • One clear move at a time: fast and precise
  • Move along a single axis: no diagonal drift
  • Sequence when it adds focus: avoid piling on
  • Use light stagger to keep dense screens readable
  • Transitions feel smooth and repeatable
  • No abrupt pops unless intentionally urgent

  • Motion answers the user’s action, then gets out of the way
  • Warm, professional micro-moments (press, select, success)
  • Calmer in core tasks: expression matches context
  • One leading direction to guide attention

  • Fade out, then in: keep text clean
  • Gentle depth/scale only when it helps hierarchy
  • Light, purposeful stagger in lists
  • No collisions elements have air

  • Transform related elements; don’t teleport
  • Let containers morph to carry context
  • The trigger and result are easy to follow
  • Changes unfold in an intelligible order

        dot.classList.remove('clicking');
        goToSlide(index);
    }, 150);
});

});

// Navigate to slide
function goToSlide(index) {
if (index === currentIndex || index >= totalSlides) return;

// Deactivate current
dots[currentIndex].classList.remove('active', 'animating');

// Activate new
dots[index].classList.add('active', 'animating');

// Reset progress
const progress = dots[index].querySelector('.pagination-dot-progress');
progress.style.animation = 'none';
setTimeout(() => { progress.style.animation = ''; }, 10);

currentIndex = index;

// Restart autoplay
clearInterval(autoplayInterval);
if (index < totalSlides - 1) {
    startAutoplay();
}

// YOUR CONTENT CHANGE CODE HERE
// e.g., show slide at index

}

// Auto-advance
function startAutoplay() {
autoplayInterval = setInterval(() => {
if (currentIndex < totalSlides - 1) {
goToSlide(currentIndex + 1);
}
}, slideDuration);
}
