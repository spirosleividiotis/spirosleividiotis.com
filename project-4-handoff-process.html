Motion design is only valuable if it ships correctly. At Qonto, I developed a handoff process that bridged design intent and engineering implementation—from initial mockups to final QA. The goal: engineers get specs they can implement, designers get results that match their vision, and users get polished motion.

2021–2025 Qonto Motion Design Process
Context

The Challenge

Static designs don't communicate motion. A Figma frame showing a modal "open" state doesn't tell an engineer how it opens—duration, easing, layering, stagger.

Early on, handoffs were verbal: "Make it slide in smoothly." Engineers would interpret this differently across platforms. iOS felt snappy, Android felt sluggish, web felt inconsistent. There was no shared language, no single source of truth.

We needed a process that was precise, visual, and implementation-ready.

Process

From Mockup to Production

01

Motion Prototyping

I start by taking static product designs and bringing them to life in Rive (or After Effects for complex sequences). Rive became my primary tool—it's lightweight, interactive, and exports directly to code.

At this stage, I'm exploring: What's the right timing? What easing feels best? Should this fade, slide, or scale? I iterate quickly, often showing multiple options to product designers or PMs.

The output is a working prototype that demonstrates the exact motion behavior—no guessing, no interpretation.

02

Applying System Tokens

Once the motion direction is approved, I map the animation to motion system tokens. If the prototype uses a 300ms fade, I replace it with motion.duration.standard. If it uses a custom curve, I check if an existing easing token works.

For custom animations that don't fit system patterns, I document the exact values and justify why they're one-off. This keeps the system clean while allowing exceptions when needed.

The goal: engineers can reference token names instead of copying arbitrary values.

03

Visual Engineering (VE) & Visual Acceptance (VA)

During VE sessions, I meet with engineers before implementation. We review the Rive prototype together, discuss technical constraints (performance, platform limitations), and align on the approach.

After implementation, we have VA sessions—visual acceptance reviews. I compare the shipped code against the original prototype. Are timings correct? Easings accurate? Layering as intended?

This isn't about nitpicking—it's about maintaining quality. Small motion discrepancies add up. A 50ms timing difference might seem minor, but it changes the feel.

04

Final Specs & QA

I provide engineers with clear, annotated specs: token references, timing diagrams, interaction states, edge cases. For Rive files, I export runtime files that engineers can integrate directly.

During QA, I test on real devices—not just simulators. Motion that looks smooth at 60fps on a MacBook might stutter on a mid-range Android phone. I check for jank, test edge cases (slow networks, interrupted animations), and ensure consistency across platforms.

The handoff isn't done until it ships correctly.

Impact

Consistent, Scalable, Shippable

This process became the standard for motion at Qonto. Engineers knew what to expect. Product designers trusted the handoff. Features shipped with polished, intentional motion—not placeholder animations.

More importantly, it scaled beyond me. The process was documented, repeatable, and teachable. Other designers could follow the same workflow. Engineers could self-serve for simple cases using the specs library.

Motion became a production-ready, predictable part of the product development cycle.
